一、从一张菜单说起
#

2024 年十一，我和朋友在莫斯科阿尔巴特大街的一家格鲁吉亚餐厅，盯着一张全是西里尔字母的菜单，陷入了每个中国游客都经历过的困境：

我们点了羊肉大包子（Khinkali，音译「辛卡利」），这是俄罗斯及东欧地区非常普遍的菜。结果端上来发现——皮是死面做的，馅料香料味极重，和我们想象中的包子完全是两回事。整顿饭吃得很难受。

但问题不在于「点错了」——问题在于我们点之前根本不知道它是什么样的。

打开 Google 翻译，拍照，翻译是翻译了。但「Khinkali」翻译成「辛卡利」之后呢？它到底长什么样？皮是发面还是死面？味道接近中餐的什么？中国人吃得惯吗？

这是一个真实的、反复出现的痛点：中国游客在海外餐厅，面对外文菜单时，缺少的不是翻译本身，而是对「这道菜到底是什么」的具象认知。

二、产品思路：不是翻译菜单，是「看懂」菜单
#

这个创意的起点来自 Andrej Karpathy 的 vibe-coding menugen。他提出了一个概念——拍菜单，用多模态模型直接把菜品图片渲染到菜单上。但我的痛点和他不完全一样：我不只是想看图，我想知道「这东西中国人吃得惯吗」。

已有的竞品（Google 翻译、各种 OCR 翻译 App）解决的是「语言转换」问题。但它们跳过了最关键的一步——帮助用户做点餐决策。

「啥菜」（WhatDish）的核心设计思路只有一句话：

拍一张外文菜单 → 输出一道「中国人看得懂的视觉菜单」——每道菜都有真实图片、中文介绍、口味标签、以及一个「中国胃适配评分」。

具体来说，输出需要包含：

  • 中文菜名 + 原文对照：知道点的是什么，也能给服务员看原菜名
  • 菜品图片：没吃过也能看到长什么样。优先搜真实照片，搜不到用 AI 生成
  • 口味标签：甜/酸/辣/咸/鲜/奶香/烟熏，一目了然
  • 中国胃评分（1-5 分）：这道菜合不合中国人口味？类似国内的什么菜？
  • 价格 + 分类：按前菜/主食/甜点分组，方便浏览

产品层面最大的取舍是：不做用户系统、不做历史记录、不做小程序。 MVP 就是一个网页，即开即用，用完即走。这是「出国点餐」这个场景的本质——低频、即兴、不需要沉淀。

三、技术选型：Software 3.0 的思路
#

在动手之前，我和 Claude（我用的 AI 编程助手）花了不少时间讨论一个关键问题：到底该怎么搭技术架构？

传统的思路是把问题拆成多步 pipeline：

OCR 引擎 → 翻译 API → 搜索引擎 → 图片聚合 → 前端渲染

每一步都要选型、对接、处理格式。这是传统软件架构的思维——AI 模型只是 pipeline 中的一环，外面包着厚厚的应用层。

然后 Karpathy 的文章给了我们一个重要启发——他称之为「Software 3.0」，意思是有一些应用应该被模型直接替代：

有些应用应该停止作为应用存在。直接把输入交给多模态模型，让它端到端地完成转换。

一个足够强的视觉模型，理论上可以一次性完成「识别菜名 + 翻译 + 理解口味 + 判断中国胃适配度」——不需要把 OCR、翻译、理解拆成三个步骤。

最终我们选择了一个折中方案：

层做什么选型
视觉理解菜单 OCR + 结构化提取MiniMax-M3（默认）/ Qwen3.7-Plus / 豆包 Seed-2.0-Pro / 任意兼容 OpenAI 的视觉模型
图片搜索搜真实菜品照片OpenSERP（开源、Docker 部署、免费无限次）
图片生成兜底搜不到时 AI 生图MiniMax image-01（默认）/ 通义万相 Wan2.7 / 豆包 Seedream
前端渲染 + 导出Next.js + Shadcn/UI + html2canvas

关键设计决策：

  • 视觉理解一步到位：一个 prompt 让模型同时完成 OCR、翻译、口味分析、结构化输出——不需要两步或三步
  • 图片搜不到就画：OpenSERP 搜百度+必应，搜不到就用生图模型画一张
  • 所有配置外置：YAML 配置文件，切换模型只改一个 provider 字段，不改代码

3.1 模型选型：可插拔比选哪个更重要
#

视觉模型这块我没有做严格的横向评测。目前默认用的 MiniMax M3，选它是因为兼容 OpenAI SDK、视觉理解能力够用、价格便宜。但说实话，千问和豆包的表现也差不多——这个场景下，模型之间的差异远小于 prompt 质量带来的差异。

核心决策不是「选哪个模型」，而是「模型随时可换」。实现方式很简单：所有模型配置集中在 YAML 里，切换 provider 只改一行：

vision:
  provider: minimax  # 改成 qwen 或 doubao 即可切换

代码层用 OpenAI 兼容 SDK（base_url + api_key + model 三个参数覆盖了千问/豆包/MiniMax/DeepSeek），不加任何 provider 专属逻辑。

一个实操经验：与其花时间看 benchmark 选模型，不如拿一张你真正遇到过的海外菜单照片，花 2 毛钱把候选模型各跑一次，看输出能不能直接用。模型能力变化太快，三个月前的结论可能已经过时了。

3.2 图片搜索 vs AI 生图：一个被低估的产品决策
#

为什么图片搜索这么重要？因为 AI 生成的菜品图对点餐决策是负资产。

测试中发现一个关键问题：让 Flux 或万相画「俄国杂拌汤」，画出来的是「一锅红汤里面有些看不清是什么的东西」。用户看了这张图，产生的不是「我要点这个」而是「这到底是什么」。

而用 OpenSERP 搜百度图片搜出来的，是真实餐厅出品的杂拌汤实拍——用户一眼就能判断「这个卖相我能不能接受」。

所以我的策略是：搜索优先、生图兜底。 搜索成本为零（OpenSERP 本地跑），生图每张约 ¥0.1-0.3。一场 12 道菜的菜单，如果全部靠 AI 画，约 ¥2-4 且可信度打折；如果大部分能搜到，成本趋近于零且可信度更高。

目前搜不到的情况主要是：小语种菜名 + 地域限定菜品（如格鲁吉亚的 Khachapuri 在百度上搜不到高质量结果）。这恰好是后续可以靠用户 UGC 填补的数据飞轮——用户上传真实菜品照片 → 数据库积累 → 后来的人直接匹配。

四、开发过程：一个下午 + 一个 AI 助手
#

起点：fork 开源项目

我们找到了一个叫 picMenu 的开源项目（433 stars），它做的事情类似——拍菜单、生成菜品图片。但它有几个问题：

  1. 只支持英文菜单，中文识别很差
  2. 强依赖 Together AI + AWS S3，国内访问不稳定
  3. 生成的图片是 AI 画的（Flux），不够真实
  4. 没有「中国胃」这个视角

所以策略是「基于它改造，而非从头写」：

  • 砍掉 S3 存储——图片在内存中以 base64 传递，用完即弃
  • 砍掉 Together AI——换用 OpenAI 兼容 SDK，支持千问/豆包/DeepSeek/自定义
  • AI 生图换成图片搜索——用 OpenSERP（百度+必应）搜真实照片
  • 加「中国胃评分」层——在 prompt 里让模型给每道菜打出 1-5 分的适配评分

踩过的坑

  1. webp 图片不触发上传：最初的 accept 列表只写了 jpg/png，忘了 webp。而且 react-dropzone 组件在 macOS 下点击不触发文件选择器。最终换成原生 <input type="file">，问题解决。

  2. 生图 API 返回 400：Qwen-Image 和万相 2.7 的参数格式不一样（size 字段一个用 "1024*1024" 一个用 "2K"），返回格式也不一样（一个返回 OSS URL 需要下载转 base64，一个直接返回 base64）。解法：根据模型名自动判断格式。

  3. 12 张图并行调用全部 429 限流：最初的实现是 Promise.all 并发 12 个生图请求，API 直接限流。改成串行调用 + 500ms 间隔 + IMAGE_GEN_MAX_COUNT 控制数量，测试时设 1 只生成一张。

实际花费

项目费用
OpenSERP 图片搜索免费（Docker 自部署）
千问视觉模型（12 道菜识别）约 ¥0.02
千问生图（12 张）约 ¥1.5
部署（本地）免费

总成本：不到 2 块钱，跑通了一整张俄语菜单。

4.1 Prompt 工程深度拆解：这 135 行才是真正的护城河
#

很多人看这个项目会说「不就是调个多模态 API 嘛，一个周末就能抄」。但如果真的抄过，会发现核心难题不是 API 调用，而是让模型稳定输出你想要的 JSON。

以下是我写完、调试完、持续改进的 vision prompt 的关键设计决策，每一个决策背后都有踩过的坑：

① 强制「是否为菜单」判断——防止模型胡说八道

用户的相册里可能有风景照、自拍、小票。如果不做判断，模型会硬生生把所有图片都「解析」成菜单。解决方案是在 prompt 开头加一段：

先判断图片内容是否是一份餐厅菜单。
如果不是菜单，直接返回 {"is_menu": false, ...}

这个看似 trivial 的判断救了无数次——用户在测试时各种图片往上传，没有这个分支判断，后端会输出各种鬼畜结果。

② 输出字段与前端 UI 同步设计——不只是「让模型输出 JSON」

每个 JSON 字段不是随便加的，背后是前端 UI 上的具体展示位置：

字段前端展示为什么需要
menu_number菜品卡片左上角编号用户拿着手机和纸质菜单对照，编号是关键锚点
name_orig卡片副标题万一想给服务员看原菜名
name_zh_orig对比展示区有些菜单自带中文翻译，标出来让用户判断可信度
name_zh卡片主标题核心信息，大字号
flavor_profile[]口味标签云一眼判断口味偏好
chinese_palate_score星级评分决策核心——要不要点
chinese_palate_note评分下方的类比「类似红烧牛肉」「接近酸辣汤」比数字更有用
tags[]筛选标签素食者/不吃猪肉/过敏原快速过滤
_source_image多图菜单来源标注用户可能拍了两页菜单，要标出来自哪张

设计原则：模型能输出的信息一次全要，前端按需展示。 不要让模型少输出（你要了 5 个字段结果发现少了第 4 个——调 prompt 再重跑），也不要让前端做二次加工（前端只能渲染，不能补知识）。

③ tags 和 flavor_profile 用封闭枚举——逼模型做选择题而非作文

这是最关键的一个决定。如果不给封闭标签列表，模型会自由发挥：

  • 好模型写出「口感绵密」「酱香浓郁」「酥脆可口」——好的但不可控
  • 差模型写出「好吃」「不错」「还行」——完全没用

封闭枚举的代价是你需要提前想清楚标签体系，但收益是前端可以直接映射到图标、颜色、筛选器，零后处理。

④ chinese_palate_score 1-5 的锚定——不能只说「打分」

如果只写「给每道菜打 1-5 分表示中国人口味适配度」，不同模型的理解差异巨大——有的模型几乎全 5 分，有的全是 3 分。

解决方案是在 prompt 中给每个分数的锚定点：

5=非常适合中国人口味，4=比较适合，3=一般可接受，2=需要适应，1=难以接受

更进一步，要求输出 chinese_palate_note（类比）——「类似红烧牛肉」这种锚定比分数更能让用户做决策。一个人看到「4 分」和看到「类似红烧牛肉」，后者的信息量大一个数量级。

⑤ 最后一行「只返回纯 JSON」——看似废话，实则救命

不加这一行，模型有 ~20% 的概率在 JSON 外面包 ```json … ``` 或加一句「好的，以下是解析结果」。虽然可以用 parseJSON() 函数兜底清洗，但让模型少犯错比让代码多容错更节约——prompt 改一行成本为零，代码加容错逻辑成本递增。

Prompt 设计的核心经验：把 prompt 当成产品说明书来写，不是当成对话来写。你是在定义 API 的 contract，不是在和 AI 聊天。每一条约束都来自于一个真实踩过的坑。

4.2 迭代方法论：如何高效调试视觉模型输出？
#

这是我自创的调试循环，至少省了我 3 小时的无效调试：

① 准备一张固定测试图 → ② 调 prompt → ③ 跑一次看输出 → ④ 对照菜单原文逐字段检查
                                       ↑                                  |
                                       └──────────────────────────────────┘
                                         如果某个字段不对：回到②，只改相关行

关键规则：一次只改 prompt 的一个维度。 如果你同时改了「格式要求」和「口味判断逻辑」，出错了无法归因。我改 prompt 的顺序是：

  1. 先确保 is_menu 判断正确（3 轮迭代）
  2. 再确保格式遵守（JSON 纯净输出，2 轮）
  3. 再调 OCR 准确率（命令模型对西里尔字母特殊关注，2 轮）
  4. 再调中文翻译质量（要求不少于 15 字的描述，1 轮）
  5. 最后调口味判断和评分（主观维度，3 轮）

总迭代次数：~11 轮，总耗时约 40 分钟。 这个时间远少于「上线后发现 bug → 修 bug → 部署 → 再测」的循环。

封版标准——什么算是「prompt 好了」？ 连续 3 张不同语言/不同排版/不同菜系的菜单照片，输出全部符合以下条件：

  • JSON 解析零错误
  • 菜名 OCR 零遗漏
  • 中国胃评分在 ±1 分内合理（人工判断）
  • 无幻觉（不编造不存在的菜）

达到这个标准后，prompt 冻结。后续的改进走 feature branch，不在生产 prompt 上实验。

六、方法论：AI 时代个人开发者的新规则
#

这次开发经历让我意识到几件事：

1. 不要从头写，从 fork 开始
#

picMenu 已经有完整的 Next.js 前端、Shadcn UI 组件、文件上传交互。我们改的是后端逻辑——模型切换、图片搜索、prompt 设计。一个人的精力有限，UI 细节不值得从头造轮子。

2. 架构做减法
#

picMenu 原有：Together AI + AWS S3 + Helicone + Next.js。我们砍到只剩 Next.js + YAML 配置。每砍掉一个外部依赖，出 bug 的概率就少一成。

3. 模型不绑死，Prompt 不硬编码
#

所有模型配置（base_url、api_key、model）都在 YAML 里，prompt 也在 YAML 里。切换千问 → 豆包改一行 provider，调整识别效果改 prompt 不动代码。配置灵活性 = 迭代速度。

4. AI 助手不是「写代码的工具」，是「产品搭档」
#

Claude 在这次项目中做的不仅是写代码——它还做了竞品调研（搜了 8 款同类 App）、架构讨论（Software 2.0 vs 3.0）、产品定位（中国胃评分这个差异化点是讨论出来的）、需求收敛（阻止我在对齐之前就写代码）。一个好的 AI 助手，应该能在你过度设计时打断你，在你遗漏时提醒你。

5. 一个人能做的事情，边界在快速移动
#

一个周末、不到 2 块钱的 API 费用、一个开源项目作为起点、一个 AI 助手作为搭档——上线了一个能处理任意语言菜单的完整 Web 应用。这在两年前至少需要一个 3 人团队 + 2 周开发 + 数千元云服务费用。

当然，我的项目还远不完美——图片搜索还没配置、导出功能待优化、移动端适配需要调整。但 80 分的执行远好过 100 分的计划。 先上线，让真实用户告诉你哪里需要改。

6. 竞争壁垒：护城河基本没有，但创意是好的
#

诚实地说，这个产品几乎没有护城河。 技术上，任何开发者花一个周末 + Claude/ChatGPT 都能搭一个功能近似的 wrapper——源码 MIT 协议，改一行 API key 就能部署。数据上，冷启动阶段菜品覆盖为零，不存在数据飞轮。

但这个创意的价值不在于「能不能做成一个独立产品」，而在于它解决了一个真实痛点，可以成为一个更大平台的产品能力。 换个角度想：大众点评海外版、携程、飞猪——这些平台天然有出境游场景和海量用户。在它们的 App 里加一个「拍菜单翻译 + 中国胃评分」功能，是一个高杠杆的体验升级，比作为独立 App 获客要容易 100 倍。

所以与其纠结「怎么建护城河」，不如想清楚两件事：

  1. 这个创意的 MVP 证明，做不做得出来
  2. 如果做出来了，是作为独立产品运营，还是作为一个案例/能力展示，融入更大的职业叙事

对我个人而言，这个项目的真正价值不一定是做成一个赚钱的产品——它是一个完整的 AI 产品从零到一的案例，方向恰好和我目前的搜索方向技术工作有交集。把这个故事讲好，本身就是一张技术名片。

7. 从 Side Project 到赚钱：三条可行路径
#

我按可行性和与能力圈的匹配度排序：

路径 A：广告 + 流量变现（低门槛、低天花板）

  • 网页端接入 Google AdSense
  • 需要日均 UV > 1000 才能产生有意义收入（约 ¥30-100/天）
  • 挑战：出境游点餐是低频场景，流量天花板低。一个用户一年出国 1-2 次，每次用 3-5 次
  • 收入潜力：乐观估计 ¥1000-3000/月（需要 DAU ~5000）
  • 匹配度：❌ 你能力圈不在这里，且流量生意和你「搜索方向运营平台」的经验不协同

路径 B：餐厅端 SaaS 收费（中门槛、中天花板）

  • 海外中餐馆/热门旅游城市餐厅付费入驻，上传真实菜单和菜品图片
  • 收费模式：¥100-500/年/餐厅，获得「认证菜单」标签（搜索结果优先展示）
  • 为什么餐厅会付费：中国游客是很多海外城市的核心消费群体。一家普吉岛的海鲜餐厅，如果能在中国游客的「啥菜」上显示认证菜单和真实菜品图，转化率远超门口拉客
  • 收入潜力：1000 家餐厅 × ¥300/年 = ¥30 万/年
  • 挑战：你需要 BD 能力——一家一家谈餐厅。或者走平台合作（携程/大众点评海外版/飞猪）
  • 匹配度：⚠️ BD 不是你的能力圈，但你的字节背景可以帮你接触到旅游平台的合作方

路径 C：被收购/并入大平台（低概率、高天花板）

  • 大众点评海外版、携程、飞猪旅行、甚至美团——都可能有出境游场景的内容需求
  • 「啥菜」的菜品数据库 + 中国胃评分体系，对这些平台有内容资产价值
  • 什么时候可能会被看上：菜品覆盖 >10 万道、MAU >10 万、或者有独特的数据资产（比如 100 个热门旅游城市的完整菜品映射）
  • 匹配度：⚠️ 不可控，不能作为主策略。但可以作为「先做着，数据积累到某个量级再考虑」的备选

我的建议：不要太早想变现。先把这个东西做出来、让真实用户用上。如果数据验证了需求，更可能的方向不是独立产品，而是作为大众点评/携程/飞猪等平台的一个功能模块——这些平台天然有出境游场景和海量用户，缺的只是这个创意和实现。路径 C（并入平台）看起来最不靠谱，反而是概率最高的终点。

8. 部署实操指南：从 localhost 到公网
#

很多人卡在这一步——本地跑通了，不知道怎么让真实用户用上。以下是最快的路径：

推荐方案：Vercel 一键部署

直接把 GitHub repo 导入 Vercel，自动识别 Next.js 项目，自动配置 HTTPS。免费额度（100GB 带宽/月、1000 次函数调用/天）对 MVP 阶段绰绰有余。3 分钟上线。

如果需要国内访问加速，后续可以迁移到阿里云轻量服务器 + nginx 反代，但 MVP 阶段不需要。

八、后续可优化项
#

这些不是承诺要做的事，只是目前能看到的改进方向，供后续迭代时参考：

功能层面

  • 配置 OpenSERP 图片搜索（Docker 已装好，config.yaml 解注释即可）——搜真实图片比 AI 画图可信度高一个量级
  • 移动端适配优化——目前 PC 优先布局，手机上字体偏小、按钮偏密
  • 多图菜单支持完善——用户拍了正反面两张菜单时，合并去重逻辑偶有遗漏
  • 导出 PNG 时样式对齐——html2canvas 在部分机型上截出的图偏移几个像素

模型层面

  • 试一下不同视觉模型在同一套 prompt 下的表现差异，找到当前最优性价比组合
  • 探索图片搜索 → AI 生图的智能切换策略（比如搜索结果的相似度低于阈值时自动走生图，而非一刀切「有图就用」）

体验层面

  • 支持用户对单一菜品打分/纠错——既是体验优化，也是数据积累的起点
  • 加 loading 骨架屏。目前等待 60 秒期间页面空白，体验不好
  • 支持语音播报菜名（给服务员看的场景——直接播放原文发音）

产品方向思考

  • 作为独立产品继续做，还是作为一个功能 demo 嵌入更大的叙事中？前者需要获客和运营，后者只需要把这个故事讲好

后记
#

如果你也对这个方向感兴趣，或者想自己 fork 一个试试：

  • 项目源码：zjdznl/whatdish（基于 picMenu 改造）

  • 核心代码不到 500 行（lib/ + app/api/parseMenu/route.ts）
