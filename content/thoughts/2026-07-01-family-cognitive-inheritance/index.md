有个程序员在 GitHub 上建了个仓库，名叫 dpan。

里面就四篇 Markdown：一篇 README，一篇关于教育的，一篇关于机遇和长期主义的小故事，还有一篇 24 万字的芒格思维模型研究。

没有 Star，没有技术含量，没有 README  badges。放在 GitHub 的海洋里，它连个水花都算不上。

但我看完之后，想了很久。

**因为这件事回答了一个问题：一个从农村走出来、白手起家的人，怎么给后代留下比钱更值钱的东西？**

## 那个仓库里的第一段话

README 是这样开头的：

> 一个家族贫穷，一定是家族里缺少领头人，就是那个高瞻远瞩，认知高，给整个家族做规划。家里长者没有能力给年轻人做规划，反而用自己低认知抹杀年轻人的未来。

他说自己完全白手起家，从家族中得到的帮助少之又少，几乎没有感知到所谓的「家族传承」。

然后他写了一句让我停顿的话：

**「我希望从我开始，后辈的发展能够越来越好，少犯我们之前犯过的错，少交我们交过的学费，少走一些人生弯路。」**

这不是一个开源项目。这是一份认知遗嘱。

他在做的事，用大白话说就是：我这辈子踩过的坑、学到的东西、悟出来的道理，在我死之前写下来。让后来人不用从头踩一遍。

## GitHub 上的「家书」

用 GitHub 写家书这件事，本身就很有意思。

你想，传统家族传承靠什么？靠口口相传，靠逢年过节的饭桌聊天，靠「你长大就懂了」。但口口相传的问题是：传着传着就丢了。爷爷跟爸爸说的，爸爸未必跟儿子说。就算说了，也可能说走样了。

**文字不会走样。Git 不会丢 commit。**

他选 GitHub 可能只是顺手——程序员嘛，写东西最习惯的就是 Markdown + Git。但这个选择恰好做对了一件重要的事：**它让这份传承变得可追溯、可更新、可复用。**

不像纸质家书会被老鼠啃掉，不像过年聊天会被打断。它是活的——想加就加，想改就改。子孙随时能看，不管人在哪个城市，不管长辈还在不在。

这可能是信息时代最务实的家族传承方式。

## 认知传承比财富传承难十倍

留下钱，相对容易。你赚到了，立个遗嘱，写个银行卡密码，完事。

但留下认知，难在哪里？

**第一，认知是隐性的。** 你知道怎么判断一个人靠不靠谱，你知道什么时候该坚持什么时候该放弃——这些判断力在你自己脑子里运行得很顺畅，但要你说出来，你发现很难。它不像「1+1=2」那样可以写成一个明确的公式。

**第二，认知需要上下文。** 你跟后辈说「要真诚」，他听听就过了。但当你说「我当年送了别人一个比特币表示感谢，这件事后来成了我人生最重要的转折点」——他记住了。因为故事提供了上下文，道理长在了场景里。

**第三，认知容易被当成说教。** 他自己也写道：「在后辈面前，我显得有些喜欢说教，喜欢讲大道理，但这种长辈往往显得有些让人厌烦。」这也是为什么他把这些东西写成文档而不是当面讲——写在纸上的道理，比从嘴里说出来的少了一层压迫感。

他在仓库里写的那篇关于教育的文章，核心观点其实就一句话：**「想让孩子做正确的事情，父母就应该做正确的事情。」** 孩子是父母的投影。父母喜欢阅读，孩子也会阅读。父母喜欢运动，孩子也会运动。父母没有担当，孩子也没主见。

这个道理不新鲜。但一个从农村走出来、亲眼见过「寒门再难出贵子」的人说出来，分量不一样。

## 为什么我觉得这件事值得聊

不是因为 dpan 写得多好。说实话，那篇芒格的思维模型研究更像是资料汇编，不是原创思考。那篇教育文章的观点也不算新颖。

**但这件事本身很有价值。** 他在做一件大多数程序员不会做的事——把自己的认知结构化、文档化、留给后来人。

我们这些人，每天写代码、写文档、写 RFC、写 postmortem。我们给公司留下了大量文字，但给自己家人留下了什么？

你写的那些设计文档，你做的那些架构决策，你踩过的那些坑，你形成的那些判断——它们是你最值钱的东西。但它们全在你的脑子里，等你退休了、老了、不在了，这些东西就没了。

**你的子女要重新踩一遍你踩过的坑，从零开始。这不是因为他们笨，是因为你什么都没留下。**

dpan 给了这件事一个模板。不是说他写的内容有多好，而是他用了一种每个程序员都能做到的方式——Markdown + Git——做了一件很少人想到要做的事。

## 你可以怎么开始

如果你也想做，不需要写 24 万字。从三件事开始：

**第一，写下你觉得最重要的三条原则。** 不是别人告诉你的，是你自己撞墙撞出来的。比如「永远不要在情绪激动的时候做决定」，比如「跟人合作之前先看他的利益结构」。

**第二，每一条原则附上一个真实的故事。** 不是「研究表明」，是「我遇到过这么一件事」。故事是认知的载体。没有故事的原则是空洞的。

**第三，把它放在一个子孙能访问到的地方。** GitHub 也好，飞书文档也好，微信收藏也好——只要它能被找到、能被更新、不会丢。

你不需要写出一本完美的书。你只需要开始写。写一条是一条。

## 最后一句话

dpan 的作者在 README 里说他希望「让遇到问题、感到迷茫的后辈有一本能够参考的人生手册」。

我觉得他想做的其实更多。他想打破那个循环——「子女走不出家族影响，不断重复上一辈生活轨迹」。

一个从农村走出来、靠自己打到今天的程序员，在 GitHub 上给后代写人生手册。这件事本身，就是那个循环被打破的证据。

**你留给孩子最好的东西，不是钱，是你犯过的错和你想通的道理。写下来。**
