2026 年，一个人说「我在做 Agent」，他可能在干这些事中的任何一件：

打开 Claude Code，写一段 prompt，看着它自动读代码、改文件、跑测试、修 bug。另一个人在用 LangGraph 画状态图，配置工具节点和条件跳转，调试为什么 Agent 卡在了第三步。第三个人用 Agno 的几行代码接上内部 API，十分钟跑通了一个 demo。第四个人手写了一个 while 循环——调 LLM、解析 tool call、执行、把结果喂回去。

他们都把这叫做「做 Agent」。

但他们做的事情差异巨大。一个在用一个**开箱即用的运行系统**（Harness），一个在**搭建**这个运行系统，一个在**快速验证**想法，一个在**从零手写**核心循环。

这些人口中的 Agent 是同一个东西吗？如果不是——那 Agent 到底是什么？

## Agent = Model + Harness

2025 到 2026 年，行业在反复争论中收敛于一个简洁的定义：**Agent = Model + Harness**。Anthropic 从长期运行 Agent 的工程挑战出发、LangChain 从开发者工具链的角度、Arize 从可观测性的视角——各自从不同路径到达了同一个公式。

**Model（模型）**负责推理。理解意图、制定计划、生成行动。它是 Agent 的大脑。

**Harness** 负责模型之外的一切。它是一套让模型能够跟外部世界交互的运行系统，具体包括：

- **编排循环**：Agent 的核心 while 循环——观察当前状态 → 模型规划下一步 → 执行工具 → 观察结果 → 重复。无论你用 Claude Code 还是手写循环，这个骨架不变。
- **工具系统**：工具注册、调用协议、工具粒度设计。模型能做什么，取决于你给了它什么工具。
- **记忆系统**：会话内的短期上下文、跨会话的持久记忆、从过往交互中学习到的经验。Agent 是在原地打转还是在持续成长，看这一层。
- **上下文管理**：当前上下文窗口里放什么、怎么压缩历史信息、什么该保留什么该丢弃。
- **Human-in-the-loop**：什么情况下需要人工确认？审批门设在哪里？人怎么覆盖 Agent 的决策？
- **安全与权限**：沙箱隔离、命令白名单、凭证管理、工具级授权。Agent 权限越大，这一层越重要。
- **用户界面**：CLI、聊天窗口、IDE 集成、API。人通过什么界面跟 Agent 交互，决定了它能嵌入什么样的工作流。
- **可观测性**：链路追踪、成本归因、性能监控。Agent 做了什么、花了多少 token、哪里卡住了。
- **评测系统**：离线评测集、在线评测、回归测试。你怎么知道这个版本比上一个好？
- **子 Agent 调度**：fork 出子 Agent 并行干活、handoff 交接任务、agent-as-tool 嵌套调用。
- **错误恢复**：重试策略、回滚机制、优雅降级。Agent 一定会犯错——错了之后怎么办。

Karpathy 有一个类比：LLM 是新的 CPU，上下文窗口是 RAM，外部数据库是磁盘，工具是驱动程序——Harness 就是整个操作系统用户态。Harrison Chase（LangChain CEO）说得更直白：「如果你不是模型，你就是 Harness。」

这个定义的价值不在它多精确，而在它暴露了一个事实：**模型怎么变，你控制不了；Harness 怎么变，你控制。** 模型的能力提升是跳跃式的——新版本发布，突然变强，但发布时间表由模型厂商决定。Harness 的能力提升是渐进式的——改一个工具接口、加一条评测用例、修一个重试逻辑，你今天就能动手，不需要等任何人的 release note。

这就引出了一个在 Agent 建设中反复出现、但很少有人明确说出来的问题：**你投入的时间，哪些能沉淀下来，哪些不能？**

## 可沉淀的与不可沉淀的

把 Harness 的各个组成部分按一个维度切开：**换了模型、换了框架之后，这项投入还在不在？**

**可沉淀的**：换了模型，评测集还在，还能跑。换了框架，知识库不需要重建。工具设计得好，换什么模型都能用。安全策略和权限模型跟具体模型解耦。这些投入有复利——今年的积累，明年还在增值。

**不可沉淀的**：模型选型今天 GPT 明天 Claude，决策有效期以月为单位。编排框架 LangGraph → Agno → Pi 生态，各领风骚。Prompt 工程跟具体模型版本耦合，换了版本就得重写。上下文管理策略随窗口大小变化而迭代——4K 时代的方案到 1M 时代要重想。

**核心命题**：可沉淀的东西决定 Agent 最终能走多远，不可沉淀的东西决定你多快到达当前的边界。

举一个具体场景：你的团队花了两个月把 Agent 从 GPT 切到 Claude，切完之后想评估效果。如果评测体系是空的——你只能靠「感觉快了」「好像准确了一点」。这两个月的工作，回报是不确定的。但如果评测体系是现成的——换完模型跑一遍评测集，35 分钟出结果，7 个指标里 5 个变好、2 个变差。这两个月的工作变成了可验证的收益。

评测体系就是典型可沉淀的东西。它不是一次性的切换成本，是资产。

这就是区分「可沉淀」和「不可沉淀」的意义——不是说不该追模型、不该试新框架，而是说，**没有可沉淀的那层地基，你在不可沉淀的东西上的所有努力，回报都是不透明的。**

## 可沉淀的四件事

### 评测：没有尺子，调参就是猜

LangChain 2026 年的行业报告给了两个数字：89% 的团队有可观测性，但只有 52% 有系统性评测。中间这 37 个百分点的差距，是生产环境 Agent 失效的重灾区。

没有评测的 Agent 开发，等于在没有测试的情况下重构代码——你可以说自己写得更好，但你没法证明。

评测有一个容易被忽略的结构性问题：**每步成功率 × N 步 ≠ 端到端成功率**。假设 Agent 完成一个任务需要 8 次工具调用，每次调用的成功率是 95%，端到端成功率只有 66%（0.95⁸）。只测单步、不测全链路——等于用及格线的零件拼出一台有三分之一概率失败的系统。

评测是典型的可沉淀资产。你今天建的评测集，明天换了模型还能跑，后天换了框架还能跑。它不依附于任何一个模型版本或框架版本。而且它有复利——每发现一个新类型的失败案例，加进评测集，这类错误就永远不会被无声地引入。

### 知识怎么组织

Agent 不能调用它找不到的知识。知识怎么结构化，直接决定了 Agent 每次能拿到什么信息。

这里说的「知识」跟后面要讲的「记忆」是两回事——很多人把它们混在一起，但它们有本质区别：

- **知识**是团队的积累——API 文档、设计规范、踩坑记录、决策上下文、评测集。它存在 Agent 外部，被 Agent 调用。
- **记忆**是 Agent 自己的积累——它在这轮会话里做了什么、上次遇到类似情况是怎么处理的、用户的偏好是什么。它存在 Agent 内部。

知识结构化不是新问题。但 Agent 时代多了一个新约束：**知识不仅要人能看懂，还要模型能精确检索。**

人看文档可以跳读、推断、自动补全上下文。一个混乱的 wiki 页面，人皱皱眉还能找到答案。但模型不行——它检索到的 chunk 是什么，它就基于什么推理。chunk 里缺了关键上下文，推理就偏。chunk 里混了多条矛盾信息，推理就乱。

这就引出了一个组织原则：**知识的单元粒度，应该匹配模型的检索和推理粒度。** 一个大而全的文档不如十个短而自包含的条目——每条讲清一件事，标题直接可检索，内容不依赖上下文就能理解。这跟给人读的文档是两种写法。

知识体系的价值不依附于任何模型或框架。你今天按这个原则整理好的知识库，GPT 能用，Claude 能用，三年后的模型一样能用。

### 工具怎么设计（以及 Skills 为什么重要）

MCP（Model Context Protocol）在 2026 年成了工具连接的事实标准——月下载量 9700 万，Linux Foundation 治理，所有主要模型供应商和 IDE 都原生支持。工具「怎么连」在快速标准化。

但工具「怎么设计」仍然是被严重低估的环节。

这里需要做一个区分——**工具（Tool）和技能（Skill）不是一回事。**

- **工具**是原子操作。「读文件」「搜索网页」「执行 shell 命令」「调用 API」。一个工具，一个函数签名，一个能力原子。
- **技能**是复合能力。它把多个工具、领域知识、执行流程、错误处理打包在一起，对外只暴露一个接口。一个技能内部可能调用了五个工具，但模型只看到一个能力入口。

举个例子：一个「发布公众号文章」的技能。内部包含读取 Markdown → 扫描隐私信息 → 转换格式 → 调用微信 API → 验证发布结果。五个步骤，每一步都可能出错，每一步都有领域知识（微信 API 的限流规则、格式转换的边界情况、隐私扫描的规则库）。如果把这些全拆成原子工具交给模型，模型需要自己编排这五步、自己理解微信 API 的 quirks、自己处理每一步的异常——每一步都在滚雪球式地增加出错概率。而一个技能把所有这些封装起来，模型只需要说「发布这篇」，技能内部搞定其余一切。

**Skills 为什么在 2026 年成为主流模式？** 三个原因：

第一，**降低模型的决策负担**。50 个原子工具意味着模型每一步都要从 50 个选项里挑，其中 40 个跟当前任务无关。10 个技能意味着模型只需要在 10 个明确的能力之间选择。选择空间越小，模型越不容易选错。

第二，**封装领域知识**。工具只告诉模型「我能做什么」，技能告诉模型「这件事应该怎么做」。前者要求模型自己发明工作流，后者把经过验证的工作流直接交给模型。模型的推理应该花在「该做什么」上，不该花在重新发明「某件事的行业标准做法」上。

第三，**技能的接口天然是模型无关的**。一个设计良好的技能——接口清晰、内部自包含、领域知识内置——换一个底层模型照样能用。因为技能的接口描述的是领域概念（「发布文章」对 GPT 和 Claude 是同一个语义），跟底层是哪个模型没有关系。**你的领域知识不会因为模型换代而过时。**

反过来，如果把领域知识散落在 prompt 和原子工具的自由组合里——那每次换模型，prompt 要重写、工具组合要重调、领域知识要从旧 prompt 里手工抢救出来。

技能封装只解决了问题的一半。另一半是：**模型怎么知道该调用哪个技能？** 一个 Agent 有 10 个技能时，用户说「帮我把上周的会议纪要整理一下发出去」，模型需要判断这是「整理纪要」技能还是「发布文章」技能还是两者的组合。这个判断——意图识别和路由——做错了，后面的技能封装得再好也触发不到。意图识别机制本身也是 Harness 层的核心组件，同样需要可沉淀的设计：路由规则怎么写、歧义怎么消解、匹配不上时怎么降级。

Skills 的设计原则——单一职责、清晰接口、领域知识内置、可独立测试——本身就是可沉淀的。你今天做出的好的技能抽象，换到下一个框架里，原则仍然成立。框架提供了技能的运行环境，但技能内部的价值不依附于框架。

### 记忆：Agent 自己的经验

记忆是 Agent 内部的积累——它做了什么、学到了什么、用户的偏好是什么。

2026 年的共识：记忆不是「加个向量数据库就完了」。它至少有三层——会话内工作记忆、近期经验、跨会话持久记忆。但三层架构只是起点。真正难的不是「怎么存」，而是**「什么时候忘」**。

人的记忆会自然衰减——不重要的东西慢慢模糊，重要的东西反复强化。Agent 的记忆不会。它只会累积。记忆量越大 → 检索空间越大 → 检索精度越低 → Agent 越来越容易拿到过时的、无关的、甚至是矛盾的信息。一个没有遗忘机制的记忆系统，最终会变成一个噪声发生器。

这就是为什么记忆系统的设计是 Harness 层的核心决策——用什么策略淘汰旧记忆、怎么判断一条记忆是否仍然有效、当新旧信息冲突时信哪个。这些不是模型能替你做的判断。

LangChain 团队在 2026 年反复强调的核心理念：「Your Harness, Your Memory」。记忆存在哪里、怎么检索、怎么更新——这些全是 Harness 层控制的。如果你用的 Agent 平台的 Harness 不透明，你的 Agent 积累的记忆也不归你控制。换了平台，Agent 失忆。

### 可沉淀 ≠ 零维护

上面把可沉淀资产的好处讲得很充分了，但这里有一个需要直面的问题：**可沉淀不等于不会腐烂。**

- **评测集会过拟合**。久而久之，你测的不是「Agent 好不好」，而是「Agent 像不像上一个版本」。评测集需要不断注入新的失败案例才有生命力，否则它会变成一个越来越宽松的自证预言。
- **知识库会膨胀和自相矛盾**。文档越积越多，新旧信息开始打架。而「模型可检索粒度」是把双刃剑——它让模型能精准找到一条信息，也让模型更容易精准找到那条已经过时的信息。
- **技能之间的交互会产生隐蔽的错误模式**。单个技能测试全绿，但技能 A 的输出作为技能 B 的输入时，类型匹配、语义漂移、副作用叠加——这些问题比原子工具的组合更难看穿。
- **记忆如果不加遗忘机制，不仅产生噪声，还会让 Agent 被过去绑架**。上一次的成功经验在新的上下文里可能是错误的选择，但 Agent 记住了「上次就是这么做的」。

可沉淀资产不会因为换模型而失效，但会因为缺乏维护而贬值。「可沉淀」说的是换了引擎不用重造车——没说车不用保养。

### 一个重要的前提：什么时候开始沉淀

还有一个容易被忽略的问题：**不是所有阶段都该优先建可沉淀资产。**

在 Agent 的产品形态还在验证、连用户要什么都不知道的阶段，花大量时间建评测体系、搞知识结构化、封装技能——可能是一种精致的资源浪费。早期需要的是快速试错，不是可复利资产。一个还跑不起来的 Agent，不值得一套完善的评测体系。

但当 Agent 开始被多人使用、开始嵌入多个场景、开始有人依赖它的输出——可沉淀资产的缺失会从「可以忍」变成「改不动」。如果此时评测体系还是空的，每次改动的风险都是不可知的；如果知识结构还是一团乱，Agent 的退化会随着规模放大。

关键不是「要不要沉淀」，是「什么时候开始」。一个粗略的判断：**当你开始害怕改 Agent 的那一天，就是你该建评测体系的那一天。**

## 不可沉淀的

### 模型选型

GPT-5.5、Claude Opus 4.8、Gemini 3.1 Pro、DeepSeek V4——2026 年的模型格局是每个月都有新选项。

选模型是重要的——但重要的是「选对了能跑多快」，不是「选错了就跑不远」。

LangChain DeepAgents 团队做过一个验证实验：模型不动，只优化 Harness（系统提示、工具配置、中间件逻辑），同一个模型从 Terminal-Bench 的第 33 名跳到了第 5 名。差出来的 28 个名次跟模型无关。

模型是 Agent 的引擎。但引擎马力大 ≠ 车跑得快——变速箱、悬挂、轮胎（也就是 Harness）决定了马力能不能转化成速度。

### 编排框架

LangGraph 用状态图抽象复杂的 Agent 工作流。Agno 走极简路线，强调快速搭建。AutoGen 以对话驱动多 Agent 协作。Pi 生态则长出了一整套多 Agent 编排框架——pi-subagents、pi-agents、pi-agent-flywheel——在终端多路复用器（Zellij、tmux）里让多个 Agent 并行协作。

Agno 官方博客最近的标题很诚实：《The Framework Wars Are Over (And Nobody Won)》。

框架不是护城河——从来不是。今天选 LangGraph，三个月后可能 Agno 更适合你的场景。选了框架之后最差的策略是把它当成长期信仰，最好的策略是反着检验：**如果下周就要换框架，哪些东西需要跟着改？** 如果答案是评测集要改格式、知识库要迁 schema、工具定义要重写——说明这些东西跟框架耦合了。如果答案是只需要把工具和技能重新注册到新框架的 runtime 里，其余一切照旧——那你的可沉淀层是扎实的。

### 多 Agent 架构

当不止一个 Agent 参与工作时，架构模式就出现了。2026 年常见的几种：

- **Router（路由）**：一个入口 Agent 判断任务类型，分发给对应的专业 Agent——「这是代码问题，给 coding agent；这是数据分析，给 data agent」。简单，适合任务类型边界清晰的场景。
- **Supervisor（监督）**：一个主管 Agent 拆解任务、分配给子 Agent、检查结果、决定是否需要返工。比 Router 多了一层质量把控，适合复杂、多步骤的任务。
- **Team（协作）**：多个对等 Agent 共享上下文、互相传递中间结果。没有中心调度者，靠协议（如 A2A）通信。灵活但调试难度高。
- **Hierarchical（层级）**：Supervisor 套 Supervisor，大规模任务逐层分解。Manus 的 Harness 架构就是典型——上层拆任务，下层执行，中间层做质检。

这些模式不是互斥的。一个系统可能入口是 Router，关键路径上跑 Supervisor，日常任务走 Team。

但模式的流行程度在变。2025 年初还在讨论「要不要用多 Agent」，到 2026 年问题已经变成了「用哪种多 Agent 架构」。明年可能又有新的模式出现。具体的架构选择是不可沉淀的——它随任务复杂度和工具生态变化。

可沉淀的是另一个东西：**Agent 之间的接口设计。** 一个 Agent 把任务交给另一个 Agent 时，传什么信息、以什么格式、期待什么返回——这个契约如果设计得清晰且模型无关，换架构模式时只需要重新编排调用顺序，不需要重写每个 Agent 的内部逻辑。

### 上下文管理策略

RAG 在上下文窗口只有 4K 的时候是标准答案。到了 2026 年，Claude Opus 4.8 上下文窗口 1M token，Gemini 3.1 Pro 也是百万级，多个模型达到或超过 1M。

策略跟着窗口变，但不只是「塞不塞得下」的问题。即使 1M 窗口能装下所有相关文档，全塞进去也有代价——推理延迟变高、token 成本飙升、以及更隐蔽的问题：信息量越大，模型越容易在「大海捞针」时注意力涣散，忽略关键事实。RAG 没有消失，它从解决「容量问题」变成了解决「精度、成本和注意力对齐问题」——帮模型在推理之前先筛掉噪声，只把最相关的信息送进窗口。

上下文管理策略会持续变化。但只要你的知识组织是清晰的、你的评测能告诉你「Agent 有没有拿到该拿的信息」，策略怎么变都不致命。

## 案例：同一个公式，两种回答

Claude Code 和 Codex CLI 都是 Model + Harness 的产物。都遵循同样的 think-act-observe 循环。都能读代码、改文件、跑命令、修 bug。

但它们走了两条不同的路：

- **Claude Code** 的 Harness 偏向缜密：交互是同步的、每一步需要人确认、上下文压缩用五阶段管线（预算缩减→裁剪→微压缩→上下文折叠→LLM 自动压缩）。设计的默认假设是「宁可慢，别出错」。
- **Codex CLI** 的 Harness 偏向速度：支持异步后台执行、沙箱内预授权、上下文策略上「用户原始消息不可触碰」——只压缩模型自己的输出，不碰用户的输入。设计的默认假设是「速度本身就是价值」。

这些差异跟模型无关。Codex 可以用 Claude 的模型，Claude Code 也不永远绑定 Opus。差异全在 Harness 的设计取舍上。

这个案例不提供「选哪个」的答案。它提供的是另一个视角：**当你比较两个 Agent 时，你比较的到底是模型，还是 Harness？** 讨论 Agent 对比的时候，第一反应通常是比模型 benchmark。但真正决定日常体验的，是 Harness 层那些看不见的决定——权限策略、上下文压缩算法、技能的粒度和设计、人机交互节奏。

## 回到开始的问题

当我们在讨论 Agent 的时候，我们到底在讨论什么？

有人在讨论模型 benchmark——GPT-5.5 比 Opus 4.8 高几个点。有人在讨论框架对比——LangGraph 和 Agno 哪个更灵活。这些讨论有价值，但它们讨论的是**不可沉淀的那一半**。它们的声音最大，因为新模型和新框架天然带着新闻属性。

但评测体系的建设、知识结构的梳理、技能封装的好坏、记忆架构的设计——这些没有发布会，没有 Hacker News 热帖。它们不发声。但它们决定了你的 Agent 最终能走多远。

区分可沉淀与不可沉淀，不是在说「别追新东西」。它是在说：**在你开始追之前，确认你的 Agent 不会因为换了引擎就要重造整辆车。**

下一次有人告诉你「XX 模型又发新版本了」的时候，先看一眼你的评测集。如果它是空的——你盯着 benchmark 数字看再久，也不会知道它在你的场景里到底是变好了还是变差了。
